Module 0

Orientation: what is inference engineering?

Training a large language model happens once. Inference — actually serving the model to users — happens billions of times, and it is where nearly all of the cost, latency, and engineering pain lives. Inference engineering is the discipline of making a trained transformer generate tokens fast, cheap, and at scale without degrading quality.

Everything in this course hangs off four numbers. Hover each card to see why it matters.

TTFT

Time To First Token. How long the user stares at a blank screen. Dominated by the prefill phase — processing the whole prompt. Compute-bound.

TPOT

Time Per Output Token. The pace of the stream after the first token. Dominated by the decode phase. Memory-bandwidth-bound.

Throughput

Tokens/second across all users. What the accountant cares about. Raised by batching, quantization, better schedulers.

Goodput

Throughput that meets the latency SLO. A server doing 100k tok/s where every request misses its deadline has goodput ≈ 0.

The single most important fact in this field: generating one token requires streaming essentially every weight of the model through the GPU's memory system. A 70B-parameter model in FP16 is 140 GB. To emit one token you must read those 140 GB from HBM. At 3.35 TB/s (an H100), that alone takes ~42 ms — regardless of how fast the compute units are. Almost every technique you will learn (batching, quantization, speculative decoding, caching) is an attack on that one bottleneck.

Module 1

Prefill vs Decode: the two-phase life of a request

A transformer serving one request runs in two radically different regimes:

  • Prefill — the entire prompt is known up front, so all its tokens are processed in parallel in one big matrix–matrix multiply (GEMM). Thousands of tokens, one pass. This saturates the GPU's compute units. Cost grows with prompt length; it sets TTFT.
  • Decode — generation is autoregressive: token t+1 depends on token t, so we must generate strictly one token at a time. Each step is a skinny matrix–vector multiply (GEMV) that still reads all the weights. The GPU's compute sits mostly idle waiting on memory. This sets TPOT.

Watch the simulation: prefill lights up all prompt tokens at once and pins compute utilization; decode crawls one token per step and pins bandwidth instead. This asymmetry is why modern servers (e.g. "disaggregated serving") often run prefill and decode on separate GPU pools.

Quick check — a user complains that long prompts make the first word slow, but streaming speed after that is fine. Which phase do you optimize?

Module 2

The Roofline Model: are you compute-bound or memory-bound?

Every kernel on a GPU is limited by one of two ceilings: the peak compute rate (FLOP/s) or the peak memory bandwidth (bytes/s). Which one binds depends on the kernel's arithmetic intensity (AI): AI = FLOPs performed ÷ bytes moved.

For a transformer forward pass with batch size B (decode: B sequences each producing 1 token), each weight byte read from HBM participates in roughly B multiply-accumulates. So AI ≈ B (in FLOPs per FP16 byte, up to a small constant). The GPU's ridge point is peak FLOP/s ÷ bandwidth — for an H100: ~989 TFLOPS ÷ 3.35 TB/s ≈ 295. Conclusion: decode at batch 1 uses well under 1% of the GPU's compute. You need a batch of ~300 concurrent tokens to leave the memory-bound region.

Module 3

The KV Cache: trading memory for compute

Attention at step t needs the Keys and Values of all previous tokens. Recomputing them every step would make generation O(n²)–O(n³) — hopeless. Instead we cache each token's K and V vectors the first time they're computed. Decode step t then only computes K/V for the one new token and attends over the cache.

The price is memory. Per token, the cache holds:

KV bytes/token = 2 (K and V) × n_layers × n_kv_heads × head_dim × bytes_per_value

Multiply by sequence length and batch size and this routinely exceeds the weights themselves. This is why GQA (grouped-query attention) exists: sharing one K/V head among a group of query heads (Llama-3-70B uses 8 KV heads for 64 query heads) shrinks the cache 8×, and MLA (DeepSeek) compresses it further into a low-rank latent. Cache dtype (FP8/INT8 KV) is another 2× lever.

Module 4

Continuous Batching: the scheduler that changed serving

Module 2 said the fix for memory-bound decode is batching. But requests don't arrive together and don't finish together. Static batching waits to collect a batch, runs it, and — crucially — a short answer must wait for the longest answer in its batch to finish. GPU slots sit idle (grey) while finished requests squat in the batch.

Continuous (iteration-level) batching, introduced by Orca and used by vLLM, TGI, and TensorRT-LLM, makes the scheduling decision every decode step: the moment a sequence emits its end-of-sequence token, its slot is handed to a waiting request. Utilization stays high, queueing time collapses, throughput typically improves by several ×.

Module 5

PagedAttention: virtual memory for the KV cache

Continuous batching creates a memory problem: you don't know how long a sequence will get, so naive systems pre-reserve contiguous KV memory for the maximum length. A request that reserves 2048 slots but generates 300 tokens wastes 85% of its allocation (internal fragmentation), and the gaps between differently-sized reservations become unusable (external fragmentation). Measurements in the vLLM paper found only 20–40% of "used" KV memory held actual data.

PagedAttention copies the oldest idea in operating systems: split the cache into small fixed-size blocks (e.g. 16 tokens), let each sequence hold a block table of pointers to non-contiguous physical blocks, and allocate a new block only when the current one fills. Waste drops to less than one block per sequence (<4%). Bonus: two sequences sharing a prompt (beam search, parallel sampling, shared system prompts) can point at the same physical blocks — copy-on-write prefix sharing.

Module 6

Quantization: fewer bits, fewer bytes, faster tokens

If decode speed = bytes ÷ bandwidth, then halving the bytes halves the time. Quantization stores weights (and optionally KV cache / activations) in low-precision integers or floats: FP16 → INT8 is 2× smaller, → INT4 is 4×. Because decode is memory-bound, this converts almost directly into TPOT speedup — the dequantize-then-multiply arithmetic is nearly free.

The core operation maps a range of real values onto a grid of 2bits levels: q = round(w / s), ŵ = q · s with scale s. The enemy is outliers: one huge weight stretches the grid, crushing precision for everything else. Real methods fight this with per-group scales (a scale per 128 weights), and smarter criteria — GPTQ (error-compensating rounding), AWQ (protect the ~1% of weights aligned with large activations), FP8 with per-tensor scaling on Hopper.

Module 7

Speculative Decoding: guess cheap, verify in parallel

Decode wastes the GPU because it processes one token per weight-stream. But we know from prefill that verifying many tokens in parallel costs barely more than one. Speculative decoding exploits this: a small, fast draft model proposes k tokens; the big target model then scores all k+1 positions in a single forward pass and accepts the longest prefix that matches what it would have sampled. A clever rejection-sampling rule makes the output distribution exactly identical to the target model alone — this is a lossless speedup.

With per-token acceptance rate α and draft length k, the expected tokens produced per (expensive) target pass is:

E[tokens] = (1 − αk+1) ÷ (1 − α)

At α = 0.8, k = 4 that's ~3.4 tokens per pass. Subtract draft cost and you get real-world 2–3× TPOT gains. Descendants of this idea — Medusa (extra decode heads), EAGLE (feature-level drafting), and n-gram lookahead — are standard in production stacks. Note the tension: raising k helps only while α is high; on hard text the draft misses and you pay the draft cost for nothing.

Module 8

Sampling & Decoding Strategies

The model outputs logits — one score per vocabulary token. Turning logits into a token is a policy choice with real latency and quality consequences:

  • Temperature T divides logits before softmax. T→0 approaches greedy argmax; T>1 flattens the distribution toward uniform randomness.
  • Top-k keeps only the k highest-probability tokens, renormalizes.
  • Top-p (nucleus) keeps the smallest set of tokens whose cumulative probability ≥ p — adaptive: narrow when the model is confident, wide when it's uncertain.
  • Min-p keeps tokens whose probability ≥ p × (max probability) — a newer variant that behaves better at high temperature.

Engineering angle: sampling runs on the critical path of every decode step, over a 100k+ vocabulary, for every sequence in the batch — production engines fuse temperature/top-k/top-p into a single GPU kernel. And greedy decoding is still not bitwise deterministic across batch sizes, because floating-point reduction order changes.

Module 9

Multi-GPU Parallelism: when one GPU isn't enough

A 405B model at FP8 is ~405 GB — no single GPU holds it. Three ways to split, with very different latency behavior:

  • Tensor parallelism (TP): split every weight matrix across GPUs; every layer ends with an all-reduce to combine partial results. All GPUs work on every token simultaneously — TPOT actually drops (each GPU streams 1/N of the weights) — but you pay a communication tax per layer, so TP wants NVLink and rarely leaves a single node.
  • Pipeline parallelism (PP): give each GPU a contiguous stack of layers; tokens flow through like an assembly line. Cheap on communication (one hop per stage boundary), but per-token latency is unchanged and naive pipelines idle in bubbles unless many microbatches are in flight.
  • Expert parallelism (EP): for Mixture-of-Experts models, park different experts on different GPUs and route tokens to them.

Real deployments compose these: e.g. TP=8 inside a node, PP across nodes.

Module 10 — Capstone

The Latency Lab: put it all together

You now own every knob in a serving stack. This calculator uses the first-order performance model that real capacity planners use:

  • TTFT ≈ 2 · params · prompt_len ÷ (TP · peak_FLOPs · efficiency) — prefill is compute-bound.
  • TPOT ≈ (weight_bytes + KV_bytes_read) ÷ (TP · bandwidth), divided by the speculative-decoding token multiplier.
  • Throughput ≈ batch ÷ TPOT; batch is capped by KV cache fitting in leftover HBM.
  • Cost = GPU-$/hr ÷ throughput → $ per million tokens.
